📈 Complexity Analysis (CLO-1) ▾

⏱️ Time Complexity

Brute Force: O(2ⁿ) — Exponential

The brute force approach enumerates all possible subsets of n items. There are 2ⁿ subsets, and for each subset we sum weights and values in O(n) time. Total: O(n × 2ⁿ). For n=20, that's over 1 million operations — impractical for large inputs.

Dynamic Programming: O(n × W) — Pseudo-polynomial

The DP algorithm fills a table of size (n+1) × (W+1). For each cell, we do constant work: either take max(include, exclude) or carry forward the previous value. This gives O(n × W) time complexity.

Greedy (Ratio): O(n log n)

Sorting items by value-to-weight ratio takes O(n log n), then a single pass to select items takes O(n). Total: O(n log n). However, this is only an approximation — it does not guarantee the optimal solution.

💾 Space Complexity

Brute Force: O(n) — Only need to store current subset being evaluated.

Dynamic Programming: O(n × W) — Stores the entire DP table of size (n+1) × (W+1). Can be optimized to O(W) by only keeping two rows.

Greedy: O(n) — Stores the items array and sorted indices.

⚠️ Why Pseudo-Polynomial ≠ Polynomial?

The DP algorithm runs in O(n × W), where W is the knapsack capacity. But W is a number, not the input size. The input size is the number of bits needed to represent W, which is log₂(W). So O(n × W) is actually exponential in the bit-length of W.

Example: If W = 1,000,000 (20 bits), then O(n × 1,000,000) is much larger than O(n × 20). This is why the algorithm is called "pseudo-polynomial" — polynomial in the numeric value of W, but exponential in the input size.

If W is bounded by a polynomial function of n, then DP runs in polynomial time. But in worst case, W can be exponentially large, making DP exponential.

📊 Algorithm Comparison Summary

Best for small n: Brute Force works fine for n ≤ 20.

Best for small W: DP is efficient when capacity W is reasonably small (e.g., W ≤ 10,000).

Best for large n and large W: Greedy gives a fast approximation, but not optimal.

Optimal choice: Use DP when W is manageable. For huge W, consider approximation algorithms or branch-and-bound.

📐 NP-Completeness Proof (CLO-5) ▾

Step 1: 0/1 Knapsack (Decision) ∈ NP

Certificate Definition: A binary string b₁b₂...bₙ of length n, where bᵢ = 1 if item i is selected, and bᵢ = 0 otherwise.

Certificate Example: For n=3 items, certificate "101" means items 1 and 3 are selected.

Verifier Algorithm (O(n) time):

function verify(certificate, items, W, V):
    totalWeight = 0
    totalValue = 0
    for i = 1 to n:
        if certificate[i] == 1:
            totalWeight += items[i].weight
            totalValue += items[i].value
    return (totalWeight ≤ W AND totalValue ≥ V)

The verifier runs in O(n) time, which is polynomial in the input size. Therefore, the decision version of 0/1 Knapsack is in NP.

Step 2: Reduction from Subset Sum

Given a Subset Sum instance (S = {a₁, a₂, ..., aₙ}, target t):

  • For each element aᵢ, create an item with weight = aᵢ and value = aᵢ
  • Set knapsack capacity W = t
  • Set value target V = t
Subset Sum
(S, t) → 0/1 Knapsack
(items: (aᵢ,aᵢ), W=t, V=t)

Forward Direction (⟹): If S has a subset summing to t, then that same subset has total weight = t ≤ W and total value = t ≥ V. Therefore, the knapsack instance is a YES-instance.

Backward Direction (⟸): If the knapsack instance has a subset with weight ≤ t and value ≥ t, since value = weight for each item (we set value = aᵢ = weight), the subset must have weight exactly t (cannot exceed) and value exactly t. This subset sums to t in the original Subset Sum instance.

Polynomial Time: The reduction creates n items in O(n) time, which is clearly polynomial in the input size.

⚠️ Why Pseudo-Polynomial DP Does NOT Contradict NP-Completeness

The DP algorithm runs in O(n × W). However, W is a numeric value, not the input size. The actual input size is the number of bits needed to represent W, which is log₂(W). Therefore, O(n × W) is O(n × 2^{bits}) — exponential in the input size.

Key Insight: If P ≠ NP (which is widely believed but unproven), then no true polynomial-time algorithm exists for any NP-complete problem. The pseudo-polynomial DP algorithm does NOT prove P = NP because its running time depends on the numeric value of W, not the bit-length of the input.

If W is bounded by a polynomial function of n (e.g., W = n²), then DP runs in polynomial time. But in the general case, W can be exponentially large (e.g., W = 2ⁿ), making DP take exponential time.

💡 Worked Example ▾

Example: Items: (2,3), (3,4), (4,5), Capacity=7, Target=7

Optimal: Take items 1 (w=2,v=3) and 3 (w=4,v=5) → total weight=6, total value=8 ≥ 7 ✓

Greedy by ratio (v/w): ratios 1.5, 1.33, 1.25 → picks 1 then 2 → weight=5, value=7 → meets target but not optimal!

⚠️ Common Mistakes ▾
❌ Mistake: Confusing decision version with optimization
✅ Fix: Decision asks "Can we achieve ≥ V?" not "What's the maximum?"
❌ Mistake: Thinking greedy works for 0/1 Knapsack
✅ Fix: Greedy by ratio fails. Counterexample below shows greedy picks items with best ratio first, missing the optimal combination.
❌ Mistake: Believing pseudo-polynomial DP proves P = NP
✅ Fix: DP is pseudo-polynomial (exponential in input bit-length). This does NOT contradict NP-completeness.
